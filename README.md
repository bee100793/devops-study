🕵️‍♂️ Server Log Analysis & Security Patch Project 26.01.17
##1. 프로젝트 개요
서버고르에 에러가 발생했어요!  
리눅스 터미널 명령어를 사용하여 서버의 에러 로그를 추출 및 백업하고, Git Branch 전략을 활용하여 안전하게 보안 패치를 적용하는 시나리오 실습입니다.

##2. 사용한 기술 & 과정 
OS: Linux (Ubuntu)

Tool: Git, GitHub, Vim

Commands: vim, grep, | (pipe), >> (append), tar, chmod, git branch/merge

Process
Step 1: 작업 공간 격리 (Git Branch)
메인 코드 보호를 위해 별도 브랜치에서 작업을 시작함.

Step 2: 로그 데이터 생성 및 분석
vim으로 기본 로그를 만들고, >>로 치명적 에러를 추가함.

Step 3: 에러 추출 및 백업 (Grep & Tar)
수많은 로그 중 'ERROR'만 추출하여 백업 폴더에 격리 후 압축함.

Step 4: 보안 스크립트 생성 (Chmod)
해결용 스크립트를 만들고, **소유자(나)**만 실행할 수 있도록 권한을 잠금.

Step 5: 본부 병합 및 전송 (Git Merge & Push)
작업 내용을 main에 합치고 깃허브에 동기화함.

Step 6: 뒤처리
임무를 다한 브랜치 삭제.


3. 트러블 슈팅 & 핵심 질문 (Troubleshooting) ✨
① 리다이렉션(>>)과 파일명 실수
문제: 로그를 추가할 때 echo "Error" >> vim server.log라고 입력하여 vim이라는 이름의 엉뚱한 파일이 생성됨.
해결: 리다이렉션(>>) 뒤에는 편집기 명령어가 아닌 **'파일 이름'**만 와야 함을 깨닫고 echo "..." >> server.log로 수정함.

② Grep과 파이프(|)의 순서
문제: 도움말에서 특정 단어를 찾기 위해 grep delete tar --help를 입력했으나 에러 발생.
해결: grep은 파일을 뒤지는 도구임. 명령어의 결과(출력물)를 검색하려면 **파이프(|)**를 통해 tar --help | grep delete 순서로 연결해야 함을 학습함.

③ 백업 시 폴더 생성과 압축의 이유
의문: "파일 하나인데 굳이 폴더를 만들고 압축까지 해야 하나?"
해결: 압축 해제 시 파일들이 쏟아지는 **'타르 폭탄(Tar Bomb)'**을 방지하기 위해 폴더에 넣는 것이 매너임을 배움. 또한 텍스트(로그) 파일은 압축 시 용량이 획기적으로 줄어들어 전송 효율이 높음.

④ Git Branch 워크플로우와 커밋 위치
의문: "새 브랜치에서 작업 후 Main에서도 add, commit을 또 해야 하나?"
해결: 작업은 오직 **새 브랜치(작업장)**에서만 수행하고 저장(commit)함. Main 브랜치에서는 merge를 통해 가져오기만 하면 된다는 Git의 핵심 철학을 이해함.

4. 학습 소감
단순히 명령어를 외우는 것을 넘어, '왜(Why)' 이렇게 사용하는지에 대한 엔지니어의 관점을 익혔습니다. 특히 **"Main 브랜치는 신성한 제품이므로, 작업은 브랜치(연습장)에서 수행하고 합친다"**는 협업의 대원칙을 완벽하게 이해했습니다. 터미널 환경에서 grep으로 데이터를 다루고 chmod로 보안을 설정하는 과정에서 CLI에 대한 자신감을 얻었습니다.
